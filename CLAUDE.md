# CLAUDE.md ─ Looop CRM ルールブック

このファイルは、本リポジトリで Claude（および人間の開発者）が UI / UX / コードを書くときに**最優先で遵守する規約**です。
要件定義は [指示書.md](指示書.md) と [docs/](docs/) を参照。本ファイルは「**どう作るか**」を定義します。

---

## 0. 大原則（これだけは絶対）

1. **このシステムは Lark のカスタムアプリとして埋め込まれる。** Lark の上に乗せて違和感がないこと。Lark を侵食しない。Lark に勝とうとしない。
2. **「AI が量産したような UI」を作らない。** 角丸 16px、グラデーション、ガラスモーフィズム、絵文字装飾、巨大ヒーローセクション、shadcn デフォルト見栄え ─ これらは即 NG。
3. **これは業務システムである。** ヒアリングに 3 分しか使えない催事現場で、汗をかいたスタッフが片手で操作する。装飾よりも**情報密度・タップ精度・読みやすさ**。
4. **個人情報を扱う。** マスキング、誤操作防止、論理削除、操作ログ。UI が「軽さ」を演出する余地は限定的。
5. **迷ったら Lark の管理画面・Larksuite 公式アプリのスクリーンショットに寄せる。**

## 1. このプロダクトのデザイン哲学

> "情報を正確に取り、同意を取り、太陽光会社へ安全に渡す" を最短で完了させる道具。

- **道具感 > 高級感。** 触っていて速い、間違えない、迷わない。
- **静か。** 賑やかな色やアニメーションは集中の邪魔。色は意味のあるところにだけ使う。
- **密度高め。** 1 画面で完結させ、スクロールを減らす。Notion ではなく Excel/Salesforce 寄りの密度。
- **抑制。** rounded、shadow、color、motion すべて控えめがデフォルト。「足したくなったら一回減らす」。
- **整列の徹底。** ベースライン、グリッド、ラベル位置、数値の桁揃え。これだけでプロっぽくなる。

## 2. Lark との関係（最重要・絶対）

このアプリは Lark のフレーム内で表示される前提。ユーザは Lark のサイドバー・ヘッダ・トーストに囲まれた状態で本アプリを見る。**Lark の中で「別のアプリだ」と認知されない**ことを目指す。

### 2.1 Lark 側に合わせる項目

| 項目 | 合わせる方針 |
|---|---|
| **プライマリカラー** | Lark Blue `#3370FF` を本アプリのアクションカラーとする。独自のブランドブルー（紫がかった青、水色など）に逃げない |
| **フォント** | Lark と同じ系統の sans-serif スタック（後述） |
| **角丸** | 4px / 6px / 8px が基準。**12px 以上は使わない**（Lark は丸くない） |
| **影** | ほぼ無し。境界はボーダー `1px solid #E5E6EB` で表現 |
| **アイコン** | 線画（line icon）、太さ 1.5px 前後、20×20 / 16×16。塗りアイコンは状態表現にのみ使う |
| **ヘッダ** | アプリ自身のヘッダは作らない／作っても 48px の薄いもの。Lark のヘッダと二重にならないこと |
| **ダークモード** | Lark がライト前提なので、当面ライトのみ。ダーク対応は Phase 3 |
| **トースト位置** | 右上ではなく **画面上部中央**。Lark と衝突しないため |

### 2.2 Lark に逆らわない（やってはいけない）

- フルブリード（画面端まで使う）の派手な hero / banner
- アプリ独自の派手なロゴをヘッダに置く
- Lark のフォントサイズを上回る巨大見出し
- 画面全体を囲うアプリ独自の border / frame
- ブラウザ全画面 modal を使った「ロックイン」演出

### 2.3 iframe 埋め込み前提の実装ルール

- `position: fixed` は**最小限**。Lark 側の overlay と競合する。
- `window.open` ではなく `target="_blank"` の通常リンク（Lark のタブ機構が処理する）。
- ファイルダウンロード（CSV）は同一タブで OK。Blob URL は revoke する。
- `localStorage` のキーは `looop_crm.*` で名前空間を切る（Lark 他アプリと衝突しないため）。

## 3. デザイントークン

`tailwind.config.ts` と CSS 変数で定義する。**マジックナンバーで色・余白・角丸を直書きしない**。

### 3.1 カラー

```ts
// tailwind tokens (HEX は基準値・Tailwind では theme.extend.colors として登録)
brand: {
  primary:    '#3370FF',  // Lark Blue. CTA / アクティブ / リンク
  primaryHover:'#2858D3',
  primaryActive:'#1E47B5',
  primarySoft:'#E8F0FF',  // Selected row, badge bg
}
text: {
  primary:   '#1D2129',  // 本文。#000 は使わない
  secondary: '#4E5969',  // ラベル、補助
  tertiary:  '#86909C',  // 非アクティブ、placeholder
  disabled:  '#C9CDD4',
  inverse:   '#FFFFFF',
}
border: {
  default:   '#E5E6EB',
  strong:    '#C9CDD4',
  focus:     '#3370FF',
}
bg: {
  base:      '#FFFFFF',  // メイン背景
  subtle:    '#F7F8FA',  // ページ背景／ゼブラ／hover
  muted:     '#F2F3F5',  // セクション区切り
}
status: {
  success:   '#00B42A',  // 契約完了、成約
  warning:   '#FF7D00',  // 審査中、要確認
  error:     '#F53F3F',  // キャンセル、撤回
  info:      '#3370FF',
  neutral:   '#86909C',  // 未提案、対象外
}
```

**ルール**

- グラデーションは禁止。例外は KPI のチャート bar にのみ許可（薄→濃の同色系）。
- ステータス色は**バッジ背景 = 該当色を opacity 12% / 文字 = 該当色 600 番台**で統一（Lark/Semi の Tag コンポーネント踏襲）。
- 「青を使えばそれっぽい」を禁ずる。**ボタンとリンクと選択状態以外には青を使わない。**
- 色だけで意味を伝えない（色覚多様性配慮 ─ アイコンや文字を併用）。

### 3.2 タイポグラフィ

```css
font-family:
  'PingFang SC',
  'Hiragino Sans',
  'Hiragino Kaku Gothic ProN',
  'Noto Sans JP',
  'Yu Gothic UI',
  'Meiryo',
  system-ui,
  -apple-system,
  sans-serif;
```

| トークン | size / line-height / weight | 用途 |
|---|---|---|
| `text-display` | 20 / 28 / 600 | ページタイトル（最大）。**1 画面に 1 つまで** |
| `text-h1`      | 16 / 24 / 600 | セクション見出し |
| `text-h2`      | 14 / 22 / 600 | カード／フィールドグループ見出し |
| `text-body`    | 14 / 22 / 400 | 本文標準 |
| `text-sm`      | 13 / 20 / 400 | 補助テキスト、テーブル本文 |
| `text-xs`      | 12 / 18 / 400 | キャプション、メタデータ、タイムスタンプ |
| `text-num`     | tabular-nums  | 金額・件数・電話番号は必ず tabular-nums |

**ルール**

- **見出しを巨大にしない。** ページタイトルでも 20px。SaaS ランディング風の 32–48px は禁止。
- 行間は詰めすぎない（line-height 1.5 前後）。
- **太字（700 / bold）は禁止、600 / semibold まで。** 重く見えるので。
- 数字は必ず `font-feature-settings: "tnum"` または `font-variant-numeric: tabular-nums`。

### 3.3 スペーシング

4px ベースの 8 スケール。Tailwind の `1 / 2 / 3 / 4 / 5 / 6 / 8 / 10 / 12` のみ使用。**`p-7` `p-9` `p-11` 等の中途半端な値は禁止。**

| 用途 | 値 |
|---|---|
| アイコンとテキストの間 | 6px (`gap-1.5`) |
| 入力フィールド内側 | 8px / 12px |
| フォーム項目間（縦） | 16px (`space-y-4`) |
| セクション間 | 24px (`space-y-6`) |
| ページ左右パディング | 24px（PC）/ 16px（モバイル） |
| カード内側 | 16px / 20px。**32px 以上は禁止**（豪華に見せない） |

### 3.4 角丸

- 入力 / ボタン / バッジ: **6px**
- カード / モーダル: **8px**
- 画像サムネ: **4px**
- 円形アバター: 50%
- **`rounded-xl` (12px), `rounded-2xl` (16px), `rounded-3xl` は使用禁止。** rounded-full はアバターとピル状ステータスにのみ。

### 3.5 ボーダー / シャドウ

- ボーダーが第一選択。`1px solid #E5E6EB`。
- シャドウは Modal / Popover / Dropdown のみ。1 種類だけ：`0 4px 10px rgba(29,33,41,0.10), 0 0 1px rgba(29,33,41,0.30)`。
- カードに `shadow-md` `shadow-lg` を撒かない。**ボーダーで十分。**

### 3.6 モーション

- 標準 transition: `150ms cubic-bezier(0.4, 0, 0.2, 1)`。
- 200ms より長いアニメーションは禁止（モーダルのフェードを除く）。
- バウンス、`spring`、`scale(1.05)` のホバー拡大、グラデーションの動きは禁止。
- ローディングは**スピナー or スケルトン**のみ。プログレスバーは長時間処理（CSV 出力等）に限定。
- `prefers-reduced-motion` を尊重する。

## 4. レイアウト原則

### 4.1 画面構造（PC / 管理画面）

```
┌────────────────────────────────────────────────┐
│ 細いトップバー（48px）  ＊Lark内なら省略可     │
├──────┬─────────────────────────────────────────┤
│      │ ページタイトル + プライマリアクション   │
│サイド│ ──────────────────────────────────────  │
│ナビ  │ フィルタ / 検索（密度高め）             │
│240px │ ──────────────────────────────────────  │
│      │ コンテンツ（テーブル or フォーム）      │
│      │                                         │
└──────┴─────────────────────────────────────────┘
```

- 最大幅は基本**設けない**（業務系は広い画面を活かす）。例外：フォーム単体ページは max-w-3xl (768px)。
- センター揃えの「ふんわり」レイアウトはダッシュボード以外で禁止。
- ページタイトル右に**プライマリ CTA を 1 つだけ**。複数ボタンを並べたい時は dropdown に。

### 4.2 余白と密度

- テーブル行高: **40px（標準）/ 32px（コンパクト）/ 48px（モバイル）**。
- リスト 1 画面に **15 行以上**見えるのが目標（PC 想定 1080p）。「カードで 1 行 80px」のような低密度 SaaS UI は不可。
- ホワイトスペースは「余裕」のためではなく「グルーピング」のために使う。

### 4.3 グリッド

- 12 カラム前提。ガター 16px。
- フォームは label-top（ラベルが入力の上）を基本。狭幅では label-left も可。
- ラベルとフィールドの幅をバラつかせない。同じセクション内では揃える。

## 5. コンポーネントルール

### 5.1 ボタン

| 種別 | 用途 | 見た目 |
|---|---|---|
| Primary | 1 画面に最大 1 個。保存・登録など主要 CTA | bg `brand.primary` / text white / hover `primaryHover` |
| Secondary | 副アクション。キャンセル等 | bg white / border `border.default` / text `text.primary` |
| Tertiary (text) | 控えめなアクション。「履歴を見る」等 | text `brand.primary` / 下線なし / hover で下線 |
| Danger | 削除・撤回 | bg `status.error` / text white。**confirm ダイアログ必須** |
| Ghost / Icon | テーブル行内の操作 | 透明背景、hover で `bg.subtle` |

- 高さ: **32px（標準）/ 28px（コンパクト）/ 40px（モバイル primary）**。
- 角丸 **6px**。pill（rounded-full）は使わない。
- アイコン+テキストの場合、アイコン 16px、テキストとの間 6px。
- グラデーション、輝き、絵文字、影 ─ すべて禁止。
- ローディング中は spinner を内側に表示し disabled に。

### 5.2 入力フィールド

- 高さ 32px（標準）/ 40px（モバイル）。
- ボーダー 1px。focus で `brand.primary` 1.5px に変える（影は付けない）。
- ラベルは入力の**上**に 4px 空けて配置、`text-sm / 500`。
- placeholder は薄いグレー（`text.tertiary`）。**例として実データを書かない**（電話番号: `09012345678` のような例示は OK だが、住所例で実在の住所を書かない）。
- 必須は label の右に小さな `*` を `status.error` で。
- エラーメッセージは入力の下に `text.xs / status.error`。**赤背景で囲わない**。
- 全角・半角の自動変換（電話番号、郵便番号）を入れる。スタッフがミスする前提。

### 5.3 テーブル（顧客一覧の中核）

- ヘッダ高 36px / 行 40px。ヘッダは `bg.subtle` / `text.secondary` / `text-xs / 600`。
- 罫線は**横線のみ**。縦線で囲わない。
- ホバーで行 bg を `bg.subtle` に変える。クリックで詳細遷移。
- 数値（金額、件数）は**右寄せ・tabular-nums**。日付は半角・等幅。
- ステータス列はバッジ。色のついた**点 + テキスト**でも可（むしろ推奨、密度が上がる）。
- ソート可能列はヘッダにアイコン。デフォルトソートを必ず決める（通常は更新日時 desc）。
- 1 行で操作（編集／削除）を出すときは**右端に icon button 1 つ**で dropdown menu を開く。ボタンを 3 つ並べない。
- 列の優先度を決め、狭画面では非優先列を隠す（折り返さない）。
- 件数が多い場合は**仮想スクロール**ではなくページネーション（50/100/200）を採用。

### 5.4 モーダル / シート

- 確認・短い入力は Dialog（中央、最大 480px）。
- 長いフォーム編集は **Sheet（右からスライドイン）**、幅 480–640px。**画面遷移を作らずに済むなら Sheet を選ぶ**（業務効率）。
- モーダルの中にモーダルを開かない。
- 閉じる動作: ✕ ボタン / Esc / 背景クリック ─ 全部効くこと。ただし入力中に背景クリックで閉じない（誤操作防止）。

### 5.5 ナビゲーション

- 左サイドナビ。アイコン + ラベル、項目高 32px。
- アクティブ: bg `brand.primarySoft` / text `brand.primary` / 左 2px の bar。
- グルーピング見出しは `text-xs / 600 / text.tertiary / uppercase` ではなく **そのまま全角の見出し**で OK（業務系日本語 UI なので）。
- breadcrumb はページ最上部に薄く（`text-xs`）置く。テーブル → 詳細の戻り導線として。

### 5.6 ステータス表示（バッジ）

- 高さ 22px、padding 2/8、`text-xs / 500`、角丸 4px。
- 例: 申込ステータスの「契約完了」= bg `#E8F8EE` / text `#00913A` / 左に 6px の dot。
- 種類が増えるので**ステータス→色のマッピングは [docs/](docs/) で一元管理し、コンポーネント側でハードコードしない**。

### 5.7 アイコン

- ライブラリ: **lucide-react**（線画・統一感あり・Lark に近い）。
- サイズ: 16 / 20 / 24px のみ。
- 装飾としてのアイコンは禁止（巨大カラフル円背景に絵文字風アイコン、等）。
- 機能を持たないアイコンは置かない。

### 5.8 カード

- ボーダー 1px、角丸 8px、bg white、padding 16–20px。
- 影なし。重ねない。並べる場合は同じ高さに揃える。
- カード内に「カードの中のカード」を作らない（区切りはボーダーか余白で）。

### 5.9 地図 UI（Google Maps 連携）

要件: 住所入力後に地図を出し、ピンを確認・修正する（[docs/05_address_googlemaps.md](docs/05_address_googlemaps.md)）。

- 地図エリアは固定アスペクト比 16:9（PC）/ 4:3（モバイル）、**角丸 6px だけ**。フローティングや影で浮かせない。
- ピンは Google デフォルトに任せず、自前 SVG ピン（`brand.primary`）を使う。確認済みは塗り、未確認は中抜き。
- 地図上のオーバーレイ UI（「ピン位置を修正」ボタン）は**右下に 1 つ**、白背景・ボーダー。半透明黒 bg は禁止。
- 「ピン確認済み」のトグルは地図の下に明示的に配置（地図上に重ねない）。

## 6. 画面パターン

### 6.1 顧客登録フォーム（催事現場・モバイル）─ 最重要画面

「3 分以内に 1 件入力できる」が KPI。**この画面の UI 設計は他のすべてに優先する。**

- **ステップ式**にする（基本情報 → 住所＋地図 → でんき情報 → 同意）。プログレスは画面上部に細い 4 ドット。
- 1 ステップは**スクロールせず親指届く範囲に収める**ことを目標。
- フォーカスは自動で次へ。電話番号入力後 11 桁で勝手に次へ進めても良い（ただし戻れる）。
- 入力途中保存は**自動（debounce 1s）+ 明示「中断する」ボタン**の二段構え。
- 必須・任意を明示（任意項目に `(任意)`）。ラベルだけで分からない項目は補助テキストで一行説明。
- 同意チェックは**必ず 2 つ独立**。デフォルトオフ。同意文言は折りたたみ可（ただし初回は展開）。
- キーボードは適切な inputmode を指定（`tel` / `email` / `numeric`）。
- iOS Safari でズームしないよう入力 font-size は 16px 以上。

### 6.2 顧客一覧

- 上部: 検索バー（フリーワード）+ フィルタ chips（担当者 / 会場 / ステータス / 同意有無 / ピン確認）+ 右に「絞り込み」ボタンで Sheet に詳細フィルタ。
- chips の選択状態は `brand.primarySoft` 背景、削除可能な × 付き。
- テーブル列の初期表示: 顧客名 / 電話 / 住所要約 / 担当 / 会場 / 申込ステータス / 同意 / 更新日時 / 操作。
- **マスキング表示**: 一覧では電話番号下 4 桁・住所は市区町村まで。詳細でフル表示。閲覧ログ記録。

### 6.3 顧客詳細

- 左 2/3 メイン（基本情報・住所・地図・申込・クロスセル・同意・履歴のタブ or アンカー） / 右 1/3 サイドパネル（担当者・最終更新・次回アクション・タグ）。
- 履歴は時系列・点でつないだ簡素なタイムライン。アバターやリッチカードは作らない。

### 6.4 太陽光連携画面

- 左テーブル: 連携対象者（フィルタ済）。
- 右パネル: 選択行のサマリ + CSV 出力ボタン。
- CSV 出力は確認 Dialog（件数・出力先会社・同意文面 version を表示）→ 出力 → トースト「N 件を CSV 出力しました」+ ログ記録。
- **同意なし顧客は UI 上から選択不可（disabled + tooltip 説明）**。色だけで伝えず、アイコンで示す。

### 6.5 KPI ダッシュボード（Phase 2）

- カードを並べる: 数値 + 前期比（小さく、色は status）。スパークラインのみ可。
- 棒グラフは色 1 色（`brand.primary`）+ ハイライト 1 色。多色虹色グラフ禁止。
- 軸ラベルは `text-xs / text.tertiary`。グリッド線は薄い `border.default`。
- 「動くダッシュボード」を作らない。リロードかフィルタ操作で更新。

## 7. モバイル / レスポンシブ

- 催事現場フォームは**モバイル/タブレット最適化を主**、PC は副。逆ではない。
- ブレークポイント: `sm 640 / md 768 / lg 1024 / xl 1280`。`2xl` は使わない。
- タップターゲットは最小 **44×44px**（Apple HIG 準拠）。
- iOS の Safe Area（`env(safe-area-inset-*)`）を考慮。固定 footer に主要 CTA を置く時は特に。
- **スワイプジェスチャーで重要操作（削除等）を発火させない。** 業務系では誤操作の方が痛い。

## 8. アクセシビリティ

- フォーカスリング: 可視。`outline: 2px solid #3370FF; outline-offset: 2px;` を消さない。
- 色コントラスト AA（本文 4.5:1、UI 部品 3:1）以上。
- ラベルと入力は `htmlFor` で関連付け。スクリーンリーダ用に `aria-required` `aria-invalid` `aria-describedby` を適切に。
- `<button>` と `<a>` を取り違えない。ナビは `<a>`、操作は `<button>`。
- 動的更新（トースト・件数変更）に `aria-live` を使う。
- 言語属性 `<html lang="ja">`。

## 10. shadcn/ui の使い方

採用するが、**デフォルトの見た目では使わない**。必ず本書のトークンに合わせて `tailwind.config.ts` と `globals.css` の CSS 変数を上書きしてから使うこと。

- `--radius` は `0.375rem`（6px）に設定。`0.5rem` 以上にしない。
- `--primary` は Lark Blue。
- `--ring` は `--primary` と同じ。デフォルトの黒系リングは使わない。
- `Button` の variant は本書の 5 種にマッピング。新しい variant を勝手に増やさない。
- 不要なコンポーネント（`Toggle`, `Toggle Group` の派手なやつ等）を漫然と入れない。

## 11. コードスタイル

- **TypeScript strict**。`any` 禁止、`unknown` で受けて narrow。
- React: Server Components 優先。Client にする時は `'use client'` の理由をコメント不要なほど明確に（state / event / browser API のいずれか）。
- ファイル分割: 1 コンポーネント 1 ファイル、200 行を超えたら分割を検討。
- スタイル: Tailwind ユーティリティを基本。`cn()`（`clsx` + `tailwind-merge`）で条件付き class。
- ロジック層と表示層を分ける（`*.actions.ts` `*.queries.ts` `*-form.tsx`）。
- **マジックストリングを定数に**（特にステータス値）。enum ではなく `as const` の union 型で。
- `/** JSDoc */` は公開 API（hooks、actions、utils）にのみ。コンポーネントには不要。
- import 順: react → next → 外部 → `@/` → 相対。
- ファイル名: kebab-case。コンポーネントは PascalCase の export を 1 つ。

## 12. 個人情報を扱う UI ルール（法令・運用）

- **マスキングが原則。** 一覧での電話番号は `***-****-1234`、住所は市区町村まで。詳細画面で初めてフル表示し、その閲覧をログる。
- **「目で見る削除」を作らない。** 削除は論理削除、UI では「アーカイブ」と表示。完全削除は管理者画面のみで二段階確認。
- **CSV 出力ボタンは管理者のみに表示**（権限で出し分け）。出すときは件数・条件・出力先会社・同意 version を必ず確認 Dialog に出す。
- **コピー禁止データ**を作らない（業務上必要なのでコピー可とする）が、コピー時にトーストで「コピーしました（操作ログに記録）」と告知し、実際にログる。
- 同意ステータスは画面のあらゆる箇所で**バッジで明示**。同意なし顧客に対するアクションボタンは disabled + 理由 tooltip。

## 14. 迷ったときの判断基準

優先順位（上から強い）：

1. **法令・個人情報保護**に反していないか
2. **Lark のデザイン言語**と衝突していないか
3. **催事現場で 3 分で入力できる**運用に資するか
4. **プロの業務システム**に見えるか（= AI っぽくないか）
5. 装飾・遊び心

迷ったら **1 つ削る**。**色を 1 つ減らす**。**角を 1 段四角くする**。**影を消す**。**フォントを 1 段小さくする**。それで成立するなら、それが正解。
